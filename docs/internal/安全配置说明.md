# 安全配置说明（内部文档）

## 1. 概述

安全过滤系统利用 LiteLLM 原生 `ContentFilterGuardrail` 和 `PromptInjectionDetection` 实现请求级内容安全。

配置文件采用 YAML 格式，存储在 `security/` 目录下。默认配置随软件发布，用户可在数据目录下创建自定义配置覆盖默认值。

## 2. 配置文件体系

```text
security/
├── security.yaml              # 主配置文件（总开关 + 各模块参数）
├── blocked_words.yaml         # 关键词黑名单
├── regex_patterns.yaml        # 自定义正则检测规则
├── injection_patterns.yaml    # 提示注入检测模式（第二期）
└── README.md                  # 用户配置说明
```

## 3. 配置加载优先级

```text
1. {用户数据目录}/security/security.yaml    （用户自定义，优先）
2. {可执行文件目录}/security/security.yaml   （发布默认，回退）
3. 都不存在 → 安全过滤自动禁用，网关正常运行
```

对于 `blocked_words.yaml` 和 `regex_patterns.yaml` 等被引用的文件，也遵循相同的优先级规则。

## 4. security.yaml 主配置字段说明

| 字段 | 类型 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- |
| `enabled` | bool | `true` | 安全过滤总开关 |
| `content_filter.enabled` | bool | `true` | 内容过滤开关 |
| `content_filter.mode` | str | `pre_call` | 拦截阶段：pre_call / post_call / both |
| `content_filter.blocked_words_file` | str | `blocked_words.yaml` | 关键词文件相对路径 |
| `content_filter.regex_patterns_file` | str | `regex_patterns.yaml` | 正则规则文件相对路径 |
| `content_filter.severity_threshold` | str | `medium` | 严重级别阈值：low / medium / high |
| `pii_masking.enabled` | bool | `false` | PII 脱敏开关 |
| `pii_masking.patterns` | list | 3条默认规则 | PII 正则规则列表 |
| `injection_detection.enabled` | bool | `false` | 注入检测开关（第二期） |
| `output_moderation.enabled` | bool | `false` | 输出审核开关（第二期） |

## 5. blocked_words.yaml 关键词规则格式

每条规则包含三个字段：

```yaml
- keyword: "国家秘密"             # 关键词（大小写不敏感）
       action: BLOCK                   # BLOCK / MASK / LOG
       description: "国家秘密保护"      # 分类说明
```

### 5.1 动作说明

| 动作 | 行为 | 使用场景 |
| :--- | :--- | :--- |
| `BLOCK` | 拦截请求，返回 400 错误 | 明确的国家秘密内容、明文凭证泄露 |
| `MASK` | 将匹配内容替换为 `[REDACTED]` | 用户自行开启的脱敏场景 |
| `LOG` | 仅记录到 security_events 表 | 凭证/密钥试探等需审计但不阻断的内容 |

### 5.2 分类设计原则

当前默认配置只保留 2 个分类：

1. **国家秘密相关**（BLOCK）：密级标识、涉密管理字段
2. **基础安全审计**（LOG/BLOCK）：凭证试探记录、明文凭证与 API Key 泄露阻断

### 5.3 添加新关键词

在对应分类下追加条目即可。关键词内可包含空格。

```yaml
- keyword: "新敏感词"
  action: BLOCK
  description: "自定义分类-说明"
```

## 6. 为什么默认不启用手机号/邮箱脱敏

本项目运行在用户自己的 Windows 电脑上，默认安全目标是：

1. 阻断明确的国家秘密相关内容外发。
2. 阻断明文凭证和 API Key 泄露。
3. 对少量基础安全试探做本地审计，便于复核。

手机号、邮箱这类普通联系方式只有在用户主动写入提示词时，网关才会看到；它们不属于这个产品的默认核心保密对象。因此默认配置中 `pii_masking.enabled=false`。

如果你的单位有额外要求，比如任何个人联系方式都不能出现在上游请求中，可以再启用该能力。

## 7. regex_patterns.yaml 正则规则格式

```yaml
patterns:
  - pattern: '\b1[3-9]\d{9}\b'   # Python 正则表达式
    pattern_name: cn_phone        # 规则名称（日志标识用）
    action: MASK                  # BLOCK / MASK / LOG
    description: "中国手机号-脱敏"   # 规则说明
```

### 7.1 注意事项

- `pattern_name` 会出现在日志的 `matched_rule` 字段中，建议使用有意义的英文名
- 正则表达式遵循 Python `re` 模块语法
- `\b` 用于单词边界匹配，避免部分匹配
- 复杂规则建议先在 Python 中测试后再写入配置

## 8. 安全事件数据结构

### 8.1 security_events 表

| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| `timestamp` | TEXT | 事件时间（ISO 8601） |
| `model` | TEXT | 请求的模型名称 |
| `event_type` | TEXT | 事件类型：keyword_block / regex_match / pii_mask / injection_detect |
| `matched_content` | TEXT | 匹配到的具体内容 |
| `matched_rule` | TEXT | 匹配的规则名称（关键词或 pattern_name） |
| `action` | TEXT | 执行动作：BLOCK / MASK / LOG |
| `request_preview` | TEXT | 请求内容预览（最多 200 字符） |
| `guardrail_name` | TEXT | 触发该事件的护栏名称 |

### 8.2 事件来源

安全事件通过 LiteLLM 的 `StandardLoggingPayload.guardrail_information` 字段获取，在 `litellm_extensions.py` 的 `async_log_success_event` 方法中提取并写入数据库。

## 9. 测试方法

### 9.1 测试关键词拦截

发送包含涉密标记的请求，确认被拦截且日志正确记录：

```bash
curl http://127.0.0.1:8765/v1/chat/completions \
  -H "Content-Type: application/json" \
       -d '{"model":"deepseek-v4-flash","messages":[{"role":"user","content":"这份材料属于国家秘密，请勿外传"}]}'
```

预期：返回 400 错误，GUI 安全过滤页显示拦截记录。

### 9.2 测试 PII 脱敏

发送包含手机号的请求：

```bash
curl http://127.0.0.1:8765/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-v4-flash","messages":[{"role":"user","content":"我的电话是13812345678"}]}'
```

预期：请求正常通过，但手机号被替换为 `[cn_phone_REDACTED]`，GUI 显示脱敏记录。

### 9.3 测试安全配置降级

删除或重命名 `security/security.yaml`，重启网关。预期：网关正常启动，GUI 安全过滤页显示"安全过滤已禁用"。

## 10. 与 LiteLLM 的集成点

安全过滤系统与 LiteLLM 的集成通过以下路径：

```text
security.yaml → SecuritySettings.load()
       ↓
SecurityGuardManager.build_guardrails_config()
       ↓
gateway_service._write_runtime_config() 注入 guardrails 段
       ↓
LiteLLM 启动时加载 guardrails 配置
       ↓
ContentFilterGuardrail 在每个请求的 pre_call 阶段执行
       ↓
guardrail 执行结果写入 StandardLoggingPayload.guardrail_information
       ↓
litellm_extensions._log_guardrail_events() 写入 security_events 表
```

完全不修改 LiteLLM 源码，所有定制在应用层完成。
