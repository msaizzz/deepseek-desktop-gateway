# 开发规范

## 1. 目标

本规范用于约束本项目后续开发、代码评审、发布和运维交接。

## 2. 基本原则

1. 所有注释和对外文档必须使用中文。
2. 本地网关是可信边界，预算、日志、认证必须在代理层保证。
3. LiteLLM 必须保留本地源码副本，不允许只依赖远程安装。
4. 优先复用现有成熟组件，避免重复造轮子。

## 3. 代码规范

1. Python 版本统一为 3.11 及以上。
2. 代码标识符可以使用英文，但注释、用户提示、文档必须是中文。
3. 修改涉及用户可见行为时，必须同时更新文档。
4. 影响预算、认证、日志结构的改动，必须补充验证步骤。

## 4. 目录规范

- `src/deepseek_gateway`：主业务代码
- `vendor/litellm`：本地 LiteLLM 源码，不随意混入业务定制
- `docs`：对外说明文档和内部规范

如需对 LiteLLM 做定制，优先在本项目侧封装；确实必须改 LiteLLM 源码时，应记录改动原因和版本差异。

## 5. 发布规范

发布前至少完成：

1. 源码可编译。
2. 关键页面可打开。
3. 本地网关可启动。
4. 预算校验有效。
5. 报表可导出。
6. 文档已同步更新。

## 6. 验证规范

每次较大改动后，至少执行以下验证：

1. 编译检查。
2. 本地启动检查。
3. 一次成功请求日志写入检查。
4. 一次超预算阻断检查。
5. 一次 CSV 或 XLSX 导出检查。

## 7. 后续扩展建议

如果未来要扩展到全研发部统一治理，建议按以下方向演进：

1. 在保持本地代理的同时增加中心汇总端。
2. 引入离线同步机制。
3. 增加更强的设备指纹校验。
4. 增加安装包、自动升级和托盘菜单。
